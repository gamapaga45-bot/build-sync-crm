# Полное руководство по экспорту в GitHub и тонкой настройке деплоя

Это руководство содержит профессиональные рекомендации по подготовке вашего репозитория и настройке CI/CD (непрерывной интеграции и деплоя).

---

## 1. Подготовка перед экспортом (AI Studio)

Прежде чем нажать "Export to GitHub", убедитесь в следующем:
1.  **Секреты:** Проверьте `.env.example`. Убедитесь, что там нет реальных ключей, а только заглушки. Реальные ключи мы добавим в настройки Render или GitHub Secrets.
2.  **Тип проекта:** Убедитесь, что в `package.json` поле `"type": "module"` присутствует, если вы используете ESM (настроено).
3.  **Порт:** Render и AI Studio используют порт 3000. В `server.ts` он зашит как константа, это правильно.

---

## 2. Экспорт и настройка репозитория GitHub

### А. Выбор типа репозитория
*   **Private:** Рекомендуется, если вы храните конфиденциальные данные о стройобъектах или планируете коммерческое использование.
*   **Public:** Подойдет для портфолио, но требует **максимальной осторожности** — никогда не пушьте `.env` файл.

### Б. Тонкие настройки GitHub (Settings)
1.  **Branch Protection Rules (Путь: Settings -> Branches):**
    *   Нажмите **Add rule** для ветки `main`.
    *   Включите **"Require a pull request before merging"**. Это не позволит случайно "сломать" сайт прямым пушем. Любые изменения должны проходить через проверку (PR).
2.  **GitHub Actions (CI - Проверка качества):**
    *   Создайте файл `.github/workflows/ci.yml`. Он будет автоматически запускать `npm run lint` и `npm run build` при каждом пуше. Так вы будете уверены, что код "собирается".
    *   *Пример содержимого:*
        ```yaml
        name: Node.js CI
        on: [push, pull_request]
        jobs:
          build:
            runs-on: ubuntu-latest
            steps:
              - uses: actions/checkout@v3
              - name: Use Node.js 20
                uses: actions/setup-node@v3
                with:
                  node-version: 20
              - run: npm install
              - run: npm run lint
              - run: npm run build
        ```

## 3. Настройка связи GitHub -> Render

При деплое на Render есть "тонкие моменты", которые сэкономят вам время и деньги:

### А. Тонкая настройка веб-сервиса (Web Service Settings)
1.  **Auto-Deploy:** По умолчанию включено (Yes). Любой пуш в `main` сразу обновляет сайт. Если вы хотите деплоить вручную, поставьте **No**.
2.  **Health Check Path:** Укажите `/api/health` (мы создали этот роут в `server.ts`). Render будет использовать его, чтобы понять, что сервер успешно запустился.
3.  **Runtime:** Выставляйте `Node` (последнюю стабильную версию, например, 20).
4.  **Region:** Для России/СНГ лучше всего выбирать **Frankfurt (EU Central)** — это обеспечит минимальный пинг (задержку).

### Б. Управление секретами (Environment Variables)
*   **Не копируйте переменные вручную:** Вы можете создать один файл `.env` на компьютере и загрузить его в Render через кнопку **"Import from .env"** в разделе Environment.
*   **DATABASE_URL:** Если вы используете базу Render, используйте внутреннюю ссылку (Internal URL), так как это быстрее и безопаснее (трафик не выходит в интернет).

---

## 4. Интеграция базы данных (Advanced)

Чтобы ваш GitHub проект был по-настоящему "боевым":
1.  **Миграции:** Используйте инструменты вроде **Prisma** или **Drizzle**.
2.  **Preload Data:** В Render в настройках БД можно указать скрипт для "накатывания" начальных данных при первом запуске.

---

## 5. Проверка после деплоя
После завершения деплоя на Render:
1.  Зайдите в раздел **Logs**. Там вы должны увидеть: `Server running on http://localhost:3000` и `Vite production server connected`.
2.  Если вы видите ошибку `Error: Cannot find module...`, проверьте, включена ли папка `src` в билд в файле `vite.config.ts`. (в нашем проекте все настроено правильно).

**Теперь ваш проект на GitHub выглядит профессионально, защищен от ошибок и автоматически доставляется пользователям!**
